## 核心方案：Segment Graph（段落图）

不要把 KMD 脚本看作一条时间线，把它看作一个**段落图（Segment Graph）**。

### 什么是 Segment

一个 **Segment** = 一段连续的、无分支的段落序列。它的特征是：**完全确定性的，可以预烘焙。**

```
Segment {
  paragraphs: KMDParagraphData[]    // 这个段的所有段落
  timeline: gsap.Timeline           // 烘焙出的主时间线（所有动画、camera、文字入场）
  entryCheckpoint: KmdSnapshot      // 进入此段时的完整逻辑状态
  exitCheckpoint: KmdSnapshot       // 离开此段时的完整逻辑状态
  duration: number                  // 确定性时长
}
```

在 Segment 内部，你可以用 `segment.timeline.seek(localTime)` **瞬间跳转到任意位置**，GSAP 会正确地插值所有跨段落动画（camera、color transitions 等等）。这解决了你和 Gemini 讨论中的"10秒camera移动跨越5个段落"的问题。

### 控制流如何映射

```
线性脚本（当前）:
  [Segment A] → [Segment B] → [Segment C]
  就是一条链，你的 bakeAll() 已经在做这件事了

分支:
  [Segment A] → branch_point
                  ├→ [Segment B1] → [Segment D]
                  └→ [Segment B2] → [Segment D]
  每条 arm 独立烘焙，各有自己的 timeline

循环:
  [Segment A] → [Segment B (loop body)] ─┐
                  ↑                      │
                  └──────────────────────┘
  loop body 烘焙一次，checkpoint 记录迭代开始状态

异步等待 (@wait_click, @wait_signal):
  [Segment A] → async_point → [Segment B]
  Segment A 的 timeline 播放到末尾就暂停
  等待完成后进入 Segment B（从 B 的 entryCheckpoint 开始）
```

### 跳转如何工作

**Segment 内跳转（最常见）：**
```
timeline.seek(targetLocalTime)  // 即时，O(1)
```
GSAP 自动处理所有插值。10秒的 camera 移动被正确地跳到中间状态。

**跨 Segment 跳转（同一条路径上）：**
```
1. 恢复目标 Segment 的 entryCheckpoint
2. 重建在场文字
3. targetSegment.timeline.seek(localTimeWithinSegment)
```
也是即时的。不需要从头回放。

**跳转到分支内部：**
```
1. 恢复分支点的 checkpoint
2. 进入目标 arm 的 Segment
3. segment.timeline.seek(localTime)
```
即时。每条分支 arm 都有自己预烘焙的 timeline。

**跳转到循环的第 N 次迭代：**
```
1. 恢复循环入口的 checkpoint
2. Warp-replay 循环体 (N-1) 次（每次只需更新逻辑状态，不渲染）
3. 进入第 N 次迭代的 Segment，seek(localTime)
```
这是唯一需要"回放"的情况，但它被限制在循环体的迭代次数内，而不是整个脚本。

## 关于 Behavior Layer 的处理

你和 Gemini 讨论的 dual-track 方向是对的。我的建议是更明确地分三类：

| 类别                    | 时间源                 | seek 行为                  | 例子                                                  |
| ----------------------- | ---------------------- | -------------------------- | ----------------------------------------------------- |
| **Entrance Anim**       | Script Time (Timeline) | `timeline.seek()` 精确插值 | 文字入场、fade in、camera move                        |
| **Persistent Behavior** | Physical Time (Ticker) | 跳转后简单重启             | shake, wave, float, pulse                             |
| **Post-entrance Chain** | Script Time (Timeline) | `timeline.seek()` 精确插值 | `.red.wait(1s).green`, `.shake(20).ease(1s).shake(5)` |

Persistent Behavior 不需要时间精确——没人在意 shake 是从第 0.3 秒还是第 0 秒开始的。跳转后直接重新启动即可。

## 你提到的核心疑问的直接回答

> "我们怎么烘焙一个分支和一个循环？"

**分支：** 每条 arm 独立烘焙为一个 Segment。你不需要烘焙"整个分支"——你烘焙每条路径。烘焙成本 = Σ(每条arm的段落数)，而不是路径的笛卡尔积。

**循环：** 循环体烘焙一次。不同迭代之间的差异只在逻辑状态（变量值）。如果循环体的动画不依赖变量，timeline 可以直接复用。如果依赖，则需要 warp-replay。

> "它们内部怎么拥有检查点，能跟主时间线怎么联动？"

每个 Segment 有自己的 local timeline。"主时间线"不再是一条全局的 GSAP Timeline，而是当前正在播放的 Segment 的 timeline。Segment 之间通过 checkpoint 衔接，不通过时间衔接。这是关键的思维转变：**Segment 之间的关系是逻辑关系（图的边），不是时间关系。**

> "进度条本身已经不存在了，只剩下代码行，更合适的方式是为它添加断点。"

完全正确。UI 应该是：
- **默认路径的 timeline**（编辑器侧边的一个微型可视化，展示当前 segment 的进度）
- **段落级断点**（点击行号 = 跳转到那个段落所在 segment 的对应时间点）
- **分支探索器**（展示当前已烘焙的分支 arm，可以点击进入）

## 实践路径

我建议分三步走，每一步都交付可用的价值：

### Phase A — Timeline 化（解决当下的跳转问题）

把当前线性脚本的 `bakeAll()` 改造为 Segment-based：
1. 整个脚本 = 一个 Segment（因为没有分支）
2. 将 `TextPlayer` 从 `setTimeout` 驱动改为构建 `gsap.Timeline`
3. Stage commands (`cam.move` 等) 也注册到同一个 Timeline
4. `seekTo()` = 恢复 entryCheckpoint + `timeline.seek(localTime)`

这一步完成后：跨段落动画的跳转就正确了，创作者不需要等。

### Phase B — Segment 图基础设施

1. 引入 `Segment` 数据结构
2. 在 parser 层识别控制流节点（如果你打算在 KMD 语法里添加 `@branch`、`@loop`）
3. `bakeAll()` 改为构建 Segment Graph 而不是 flat list
4. 跳转逻辑走 Segment Graph

### Phase C — 交互式控制流

1. 实现 `@wait_click`、`@branch`、`@loop` 语法
2. `SignalRegistry` 跟踪异步事件状态
3. 循环跳转的 warp-replay
4. 编辑器的分支探索 UI

## 总结

你不需要"为 DSL 做一个完整的 debugger"——你需要的是一个**段落级别的检查点系统 + 段落内的时间线驱动**。这两者的组合已经足够强大：

- 线性部分 → timeline seek（即时）
- 分支 → 独立烘焙每条 arm（即时）
- 循环 → warp-replay 迭代（快速，有界）
- 异步等待 → checkpoint + resume（Segment 边界自然处理）

最关键的心智转变是：**放弃"全局时间线"的概念，拥抱"Segment 图 + 局部时间线"。** 时间是 Segment 内部的概念，Segment 之间靠逻辑状态（checkpoint）衔接。
