# KMD 1.6 Phase B Feature Plan

> 状态：Draft
> 最近更新：2026-05-11
> 范围：Segment Graph、StateStore、Control Flow、B0 语法演进，以及随功能推进同步偿还的 execution 技术债

## 1. 定位

Phase A 与 Phase B Prep 已经完成 parser、layout、execution、stage、diagnostics 和 metadata 的主链路准备。Phase B 不再是“继续打地基”，而是开始引入用户可见能力：

- `+` 并发链、续行符、文本插值
- `StateStore` 与表达式求值
- `@ if / @ loop / @ tag / @ jump / @ wait`
- `SegmentGraphPlan` 与跨 Segment seek

但 Phase B 也不能把新功能继续塞进旧热区。每个功能工作包都必须绑定一个债务收口目标，避免 `TextPlayer`、`SegmentBuilder`、`EffectProcessor` 和 core/store 耦合继续膨胀。

## 2. 设计原则

- 新语法先进入 `DocumentAST / DocumentSemanticIR`，不要直接投影成 legacy token 或 layout stream item。
- Segment 内部仍是确定性 `gsap.Timeline`；Segment 之间的边由 graph runtime 按 state/control-flow 求值。
- `LayoutStream` 只作为 layout lane 内部表示，不承载 stage、playback、control-flow 总线职责。
- 新 execution 语义必须进入 plan/cue，不新增 legacy `TextPlayer.play()` 分支。
- core 层不新增 `useEditorStore()`，后续改为 host callback / adapter。
- stage/effect/layout 行为优先读 metadata，不新增硬编码 command set。

## 3. 目标管线

```text
Source
  -> DocumentParser
  -> DocumentAST
  -> AstNormalizer
  -> SemanticLowerer / ControlFlowLowerer / StateLowerer
  -> DocumentSemanticIR
  -> EffectMiddleware / LayoutMiddleware / StageMiddleware
  -> ParagraphExecutionPlan
  -> SegmentExecutionPlan
  -> SegmentGraphPlan
  -> GraphPlaybackRuntime
```

第一轮不要求一次落齐所有层。实现上可以保留当前 `ParagraphIR` 与 compat projection，但 Phase B 新能力必须有 document-level 语义节点，不再伪装成段落内普通命令。

## 4. 工作包

### B0. Syntax Frontend

功能：

- `+` 并发链：同一 token 上多条独立 command chain。
- 行尾 `\` 续行：复杂 command chain 可跨行书写。
- `{var.xxx}` 文本插值：正文中引用变量或 state 值。

偿债：

- 把并发链表达为 `ParallelCommandChainAst` / chain node，不在 `EffectProcessor` 中二次猜测。
- 将续行归入 `AstNormalizer` 或 document parse 预处理，不复活旧 scanner 分支。
- 插值生成独立 `InterpolationNode`，不把 `{var.*}` 当普通字符串延后碰运气。

验收：

- parser fixture 覆盖三种新语法。
- compat projection 对未启用新语法的脚本零行为变化。

### B1. StateStore And Expressions

功能：

- `StateStore { get, set, snapshot, restore }`。
- 极简表达式求值：字面量、变量、算术、比较、逻辑。
- `@ set gold = gold + 5` 与 inline `state.set(gold, gold + 5)`。

偿债：

- `RuntimeValueResolver` 成为 state / var / marker 的统一解析入口。
- `Checkpoint` 增加 `state`，不再只保存 stage/layout。
- 当前 frontmatter `variables` 写入 `layout.globalMarkers` 的路径要收口为兼容投影，避免 state 和 marker 双真相。

验收：

- seek 前后 state 可恢复。
- `@ set` 不依赖 stage preset 伪装执行。

### B2. Document Control Flow

功能：

- `@ if / @ elif / @ else / @ end`
- `@ loop / @ while`
- `@ tag / @ jump`
- `@ wait click`，`@ wait signal name` 先预留 signal 形状

偿债：

- 新增 `DocumentParser` 或 document-level parse entry，段落 parser 不继续负责文档结构。
- `ControlFlowLowerer` 输出结构化 control nodes，不把控制流压进 `paragraph.globalEffects`。
- 控制流 diagnostics 进入统一 diagnostics collector。

验收：

- 能生成 document-level control tree。
- 第一阶段可只支持 syntax + graph plan，不急于完整 runtime 行为。

### B3. SegmentGraphPlan

功能：

- `SegmentGraph { nodes, edges, defaultPath }`。
- 每个 graph node 持有一个烘焙好的 deterministic segment。
- edges 支持 condition、default、jump target、wait gate。

偿债：

- `SegmentBuilder` 降级为“单 Segment builder”，不再承担 graph 切割。
- 抽出 `SegmentGraphBuilder`，负责 paragraph group 切割、edge 生成、tag 索引。
- stage tween conflict 与 in-flight record 继续保留，但 planner 先产出 diagnostics，runtime trim 只做兜底。

验收：

- 线性脚本能被表示为只有 default edges 的 graph。
- 原有 single-segment playback 行为保持一致。

### B4. Graph Playback And Seek

功能：

- 同 Segment 内 seek 继续使用 `timeline.seek(localTime)`。
- 跨 Segment seek 恢复目标 `entryCheckpoint`，再 seek 到本地时间。
- 条件分支 seek 默认沿 `defaultPath`。
- loop/while seek 先采用入口 checkpoint + warp replay。

偿债：

- 抽出 `GraphPlaybackController`，不要继续把 graph 行为塞进 `ScriptPlayer`。
- `ScriptPlayer` 收缩为 load/build/play facade。
- core 到 editor store 的时间、行号、marker 更新改为 callbacks。

验收：

- seek、play、pause API 对 editor 仍稳定。
- 新 graph runtime 不调用 legacy `TextPlayer.play()`。

### B5. Execution Debt Closure

功能目标不是新增语法，而是为 Phase B 中后段降低维护痛感。

任务：

- 标记 legacy `TextPlayer.play()` 为不再扩展路径；达到覆盖条件后删除或移到 compat module。
- 将 `TextPlayer.buildTimeline()` 中剩余 chain 分流继续前移到 execution plan / EffectMiddleware。
- 拆 `SegmentBuilder.applyStageConfigs()` 为 stage timeline coordinator。
- `KineticChar` 不新增语义字段，逐步只保留 display host 与 transform state。

验收：

- 新能力只消费 `ParagraphDisplayAssembly`、execution plan、graph plan。
- `KineticChar.stageInstructions / visualEffects / timingSugars` 仍仅作为 deprecated mirror。

### B6. Editor And Diagnostics Integration

功能：

- Monaco 控制流折叠与 segment boundary 标记。
- timeline markers 展示 graph default path。
- Inspector 显示 command metadata、state snapshot、graph edge diagnostics。

偿债：

- editor UI 通过 adapter 读取 runtime snapshot，不让 core import Pinia。
- diagnostics/audit 继续走统一总线，不新增孤立 `console.log` 调试路径。

验收：

- Phase B 新 diagnostics 能在 editor 和 console sink 中同时观察。

## 5. 推荐实施顺序

1. B0.1 `+` 并发链 syntax-only + parser regression。
2. B0.2 续行与 B0.3 插值 AST，暂不接复杂 runtime。
3. B1 StateStore + expression + `Checkpoint.state`。
4. B2 document control-flow syntax tree。
5. B3 线性 `SegmentGraphPlan`，先把现有脚本包进 graph。
6. B4 graph runtime 接 `@ tag / @ jump / @ if` 的最小行为。
7. B5/B6 穿插推进，不能等功能全部完成才还债。

## 6. 非目标

- 不在 Phase B 初期抽 `apps/editor/src/core` 到 `packages/core`。
- 不实现 Phase C 的完整 `SignalRegistry` 与游戏化实时 Segment。
- 不启动 v1.7 plugin hook 系统，只保持 metadata 与 registry 入口可扩展。
- 不把 Android Reader 集成进当前 runtime 构建链；只保证未来可消费稳定产物。

## 7. Gate

每个工作包至少满足：

- `pnpm build`
- `pnpm test:parser`
- `pnpm language:check`
- 关键样例脚本人工回归
- 新语法必须有 parser fixture 或等价回归脚本

B3 之后需要补 execution/graph 回归脚本，不能继续只依赖 parser test。

## 8. Open Questions

- `state.*` 与 `var.*` 是否长期保留双命名空间，还是 `var.*` 降级为 state alias。
- `+` 并发链是否允许跨 lane 混写 stage/layout/effect，还是第一版限制为 visual/effect chain。
- `@ loop N` 当 `N` 是表达式时，是编译时求值还是运行时 edge 计数。
- graph seek 对非 default path 的 UI 预期：近似跳转、路径选择器，还是禁止直接 seek。
- legacy `TextPlayer.play()` 的删除条件：需要哪些样例和自动测试覆盖后才安全。
